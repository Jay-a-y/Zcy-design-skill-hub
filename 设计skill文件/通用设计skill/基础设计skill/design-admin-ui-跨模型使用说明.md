# 后台原型与视觉规范 Skill：跨模型使用说明

## 1. 能否在其他模型使用

可以，但分为两种兼容层级：

1. **支持 Skill/Agent Skill 的平台**：保留整个 `design-admin-ui` 文件夹，安装到平台规定的用户级或项目级 Skill 目录，再调用 `design-admin-ui`。这类平台可以利用 `SKILL.md` 的名称、描述和渐进式引用。
2. **不支持 Skill 的普通模型或聊天产品**：把 `SKILL.md` 作为系统提示或项目规则，并按任务上传相关 `references/` 文件。模型仍可执行设计流程，但不会自动识别 `$design-admin-ui`，也不一定能生成、运行或测试代码。

模型至少需要读取 Markdown 文件。要输出可点击 Demo，还需要文件写入、代码执行和浏览器预览能力。纯图片模型只能充当渲染环节，不能独立执行完整 Skill。

## 2. 分发包内容

```text
design-admin-ui/
├── SKILL.md                 # 总入口与执行规则
├── agents/openai.yaml       # Codex 展示与默认调用信息
├── references/              # 原型流程和分组件规范
├── assets/screenshots/      # 截图证据
└── assets/source/           # 标准组件源码参考包
```

不要只分发 `SKILL.md`。完整的视觉规则、组件选型和原型交付流程位于 `references/`。

## 3. 通用安装方式

### A. 支持 Skill 的智能体平台

1. 下载并解压推广包。
2. 找到其中的 `design-admin-ui` 文件夹。
3. 将整个文件夹复制到平台文档指定的用户级或项目级 Skill 目录。
4. 重启应用或新建会话，使平台重新扫描 Skill。
5. 在新会话中调用 `design-admin-ui`；若平台支持 `$` 语法，可使用 `$design-admin-ui`。
6. 先用一个单页需求测试文件读取、代码生成和预览能力，再推广到团队。

具体目录和调用语法以各平台当前官方文档为准，不要假设不同平台与 Codex 使用相同路径。

### B. 支持项目规则但不支持 Skill 的平台

1. 把 `SKILL.md` 内容放入该平台的项目规则、系统指令或持久上下文。
2. 保留 `references/` 在项目中，并告诉模型只读取与当前任务相关的文件。
3. 将下面的“通用启动提示”设为项目默认提示。
4. 如果平台不能读取文件夹，先上传 `SKILL.md`、`prototype-workflow.md`、`foundations.md`，再按页面类型上传一至两个组件规范。
5. 明确要求模型输出可编辑源文件，而不是只返回说明文字。

### C. 普通网页聊天模型

1. 新建对话或项目知识库。
2. 上传 `SKILL.md` 和必要参考文件。
3. 粘贴“通用启动提示”。
4. 粘贴产品需求模板。
5. 如果平台不能创建文件，让模型输出完整 HTML/React 代码，再交给能运行代码的工具验证。
6. 将运行截图和问题继续回传给模型迭代。

### D. API 或企业内部智能体

1. 把 `SKILL.md` 作为基础系统指令。
2. 根据任务分类检索并注入相关参考文件，避免一次加载全部规范。
3. 为智能体提供文件系统、代码执行和浏览器截图工具。
4. 将源码、截图、控制台日志和验收结果作为结构化输出。
5. 对外部素材、内部数据、网络请求和生产写入设置权限边界。

## 4. 通用启动提示

```text
你将使用 design-admin-ui 完成后台产品设计任务。

执行前：
1. 完整阅读 SKILL.md。
2. 阅读 references/prototype-workflow.md 和 references/foundations.md。
3. 根据任务类型，只加载相关组件参考文件。
4. 优先复用现有项目技术栈、设计令牌和组件库。

执行要求：
- 将需求转成最小完整用户流程和页面清单。
- 输出可编辑的原型或可运行 Demo，不要只给文字建议。
- 使用真实、合理的中文业务数据。
- 实现主要交互，并覆盖加载、空、无结果、校验、错误、禁用、无权限和成功状态。
- 检查响应式、键盘操作、内容溢出和可访问性。
- 最终提供源文件、启动方式、验收截图、关键假设和未覆盖范围。
```

## 6. 推荐调用示例

### 产品经理

```text
使用 design-admin-ui，把下面需求制作成可点击后台 Demo。

项目：供应商审核
用户：采购运营
任务：筛选报名供应商，查看资料，批量通过或驳回
页面：审核列表、详情抽屉
要求：包含筛选、分页、批量操作、驳回确认、加载、空状态和操作反馈
交付：可运行源码、启动方式和桌面端验收截图
```

### 设计师

```text
使用 design-admin-ui，为“合同履约风险看板”输出高保真原型。

需要：看板主页、筛选展开态、风险详情抽屉、空状态和错误状态。
沿用现有蓝白后台风格，提供可编辑源文件和 1440px 桌面端截图。
```

### 研发或 AI 编程平台

```text
使用 design-admin-ui 在当前项目实现供应商审核页面。
保留现有框架和组件库，不重建基础设施。
完成后运行项目，测试主要交互和控制台错误，并输出修改文件与验收截图。
```

## 7. 验收清单

- 页面能否回答“我在哪里、看到什么、下一步做什么”。
- 主流程是否可以真实点击完成。
- 主次操作、危险操作和反馈层级是否清楚。
- 表格、表单、抽屉和弹窗是否符合规范。
- 是否覆盖加载、空、无结果、错误、无权限和成功状态。
- 中文长文本、大金额、日期、缺失字段是否正常展示。
- 窄屏、滚动、粘性区域和浮层是否遮挡内容。
- 是否存在无效按钮、假链接或控制台错误。
- 是否交付可编辑源码、启动方法和截图。

## 8. 推广与版本管理

- 团队只维护一个主版本，避免成员自行复制后长期分叉。
- 每次更新保留 `SKILL.md` 名称不变，并记录内部版本号或发布日期。
- 先用 3–5 个真实需求分别在目标模型上试运行，比较页面完整度、交互、代码质量和工具调用能力。
- 为每个平台保存一条经过验证的启动提示，但不要修改核心视觉规范。
- 外部传播前确认政采云品牌、截图和内部业务组件是否允许公开；标准组件源码包本身包含 MIT 许可文件，但这不自动覆盖其他素材。
