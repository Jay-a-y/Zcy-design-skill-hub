# 云随 UDesign 设计 Skill 使用说明

## 1. 主要给谁使用

这份 Skill 的主要日常使用者是：

- 正在做云随需求的产品经理。
- 正在开发或验证云随需求的研发人员。

日常使用时，不需要产品经理或研发理解完整设计规范，也不需要自行判断 L0～L3、寻找 Figma 节点、选择 UDesign 组件或决定 quick-use 是否启用。

最常见的使用方式只有一句话：

```text
把云随项目设计 Skill 和本次需求文档、业务诉求或测试数据一起交给 AI，让 AI 按 Skill 生成 Demo。
```

## 2. 这份 Skill 已经包含什么

设计师已经在 Skill 中提前沉淀：

- 云随的业务阶段、服务和角色权限。
- 首页、上传、起始、等待、排队、Loading、失败、超时、结果和历史态等成熟页面模式。
- 正式 Figma 样板及其适用范围。
- UDesign 基础视觉交互组件和框架映射。
- quick-use 的使用边界。
- AI 结果、证据定位和“AI已回填”规则。
- P0/P1/P2 设计准入标准。
- 单 HTML Demo 的默认交付规则。

因此，产品经理和研发不需要在每次需求里重新描述这些规则。

## 3. 最简使用流程

```text
云随 Skill + 需求文档/业务诉求/测试数据
                    ↓
AI 自动匹配业务规则、正式样板和成熟页面模式
                    ↓
生成可直接打开的页面 Demo
                    ↓
产品/研发验证流程和内容
                    ↓
复杂设计缺口或重复问题反馈设计师
```

### 第一步：准备材料

至少提供：

1. 云随项目设计 Skill 文件夹。
2. 本次需求文档、需求描述或原型说明。

可按需求补充：

- 测试文件、招标文件或业务数据。
- 现有 Demo 或代码。
- 本次必须覆盖的页面或流程。
- 期望交付形式，例如单 HTML、现有代码改造或设计评审。

需求文档可以保持产品经理或研发平时使用的格式，不需要为了 AI 重新整理成专业设计 Brief。

### 第二步：把材料交给 AI

直接使用：

```text
读取云随项目设计 Skill 和本次需求文档，根据需求生成一份完整 Demo。

请自行复用 Skill 中已有的业务规则、正式 Figma 样板、成熟页面模式和 UDesign 映射；补齐适用的等待、排队、Loading、结果、失败、超时、不可用、无权限和历史状态。

默认交付一份可以直接双击打开的单 HTML，不要生成工程脚手架。如果需求包含 Skill 尚未覆盖的复杂核心页面，不要自行设计，列出需要设计师补充的设计缺口。
```

### 第三步：产品或研发验证

拿到 Demo 后重点确认：

- 功能流程是否符合需求。
- 页面文案和业务数据是否正确。
- 角色和权限是否正确。
- 开始、返回、取消、重试和完成后的去向是否正确。
- 是否覆盖本次真正需要的状态。

产品或研发不需要检查具体使用了哪个 UDesign 组件；这部分由 AI 自检和设计师抽查。

### 第四步：反馈问题

如果只是本次需求内容错误，直接让 AI 修改 Demo。

如果发现以下问题，反馈给设计师：

- 缺少关键页面样板。
- AI 多次误解同一页面模式。
- 新增复杂核心流程或高风险操作。
- 新增角色、项目阶段或权限规则。
- Figma 正式样板发生变化。
- 同类问题在多个需求中重复出现。

设计师再判断是否需要更新云随 Skill。

## 4. AI 会自动完成什么

产品经理或研发把 Skill 和需求交给 AI 后，AI 应自动：

1. 从需求文档中识别角色、阶段、服务、目标和主要操作。
2. 读取 Skill 中对应的业务规则。
3. 自行定位已收录的正式 Figma 节点和成熟页面模式。
4. 判断需求是否落在 Skill 已沉淀范围内。
5. 按正式 Figma 复用已有页面布局和项目视觉。
6. 使用 UDesign 实现基础组件与交互。
7. 只对非核心缺失部分使用 quick-use。
8. 补齐适用的异常、权限和历史状态。
9. 生成可运行 Demo。
10. 执行 P0/P1/P2 自检。

这些是 AI 的执行步骤，不是产品经理或研发需要提前完成的任务。

## 5. 产品或研发需要提供什么

### 通常已有的信息

- 需求要解决什么问题。
- 哪类用户使用。
- 发生在什么业务流程中。
- 页面需要展示什么内容。
- 用户要进行什么操作。
- 完成后应该发生什么。

### 最好明确的信息

- 本次是生成全流程还是单个页面。
- 是否需要使用真实业务文件或测试数据。
- 是否需要模拟 Loading、失败和结果。
- 交付一份单 HTML，还是修改已有代码。
- 哪些内容必须由产品确认，不能由 AI 假设。

### 不需要提供的信息

- 不需要判断 L0、L1、L2 或 L3。
- 不需要重新整理 Skill 已收录的 Figma 地址。
- 不需要指定 Button、Upload、Drawer 等组件。
- 不需要决定 UDesign 与 quick-use 的使用顺序。
- 不需要先写 P0/P1/P2 评审结论。

## 6. 适合直接生成的需求

- 在已有首页或二级页模式上替换真实业务内容。
- 采购文件提取、检测、章节推荐、围串标审查和辅助评标的成熟流程。
- 上传、等待、引擎就绪、排队、Loading、失败、超时、不可用和历史态。
- 已有结果页框架内的信息调整。
- 宿主页面中的免费使用提示和“AI已回填”标记。
- 基于真实招标文件或测试数据生成完整流程 Demo。

## 7. 需要回到设计师的需求

以下情况不应让 AI 自由设计核心页面：

- 新增业务阶段。
- 新增角色工作台或大幅调整权限体系。
- 新首页骨架或新的核心导航方式。
- Skill 中没有样板的复杂核心页面。
- 新的高风险或不可逆操作。
- 新的风险判断机制、关键结果决策或复杂证据交互。
- 正式 Figma 与需求存在实质冲突。

AI 可以先生成已覆盖的成熟部分，但应把未覆盖部分列成设计缺口，不得自行把临时方案当作云随正式规则。

## 8. 设计师在 Skill 交付前做什么

L0～L3 属于设计师建设和维护 Skill 时使用的治理方法：

| 分级 | 设计师处理方式 |
| --- | --- |
| L0 | 确认基础组件和局部调整可由 UDesign 覆盖 |
| L1 | 确认已有成熟页面模式可以复用 |
| L2 | 先确认结构、状态和关键组合，再沉淀到 Skill |
| L3 | 先手工设计核心 Figma 样板，确认后再更新 Skill |

正确顺序是：

```text
设计师判断模式复杂度
→ 必要时手工完成核心 Figma 页面
→ 把业务规则、正式样板和组件映射沉淀进 Skill
→ 验证后把 Skill 交给产品经理和研发
→ 产品/研发用 Skill + 需求直接生成 Demo
```

因此，L0～L3 不应出现在面向产品经理和研发的日常使用步骤里。

## 9. AI 内部设计来源顺序

产品或研发不需要操作这一部分，但 AI 必须遵守：

```text
本次明确需求
→ 正式 Figma
→ UDesign
→ 云随成熟页面模式与业务规则
→ quick-use 非核心缺失兜底
```

- 正式 Figma 决定已有页面的最终布局和项目视觉。
- UDesign 提供基础视觉交互组件和框架实现。
- quick-use 只补齐前两者和成熟模式均未覆盖的非核心部分。
- 产品 Demo 只提供功能和流程，不作为视觉依据。

## 10. 默认交付形式

用户说“生成 Demo”时，默认应交付：

- 一份单 HTML 文件。
- 双击即可打开。
- CSS、JavaScript、图标和必要资产全部内联。
- 不需要安装依赖或启动服务。
- 不创建 React、Vite、Vinext 等工程结构。

只有用户明确要求工程化代码时，才创建项目工程。

## 11. AI 交付时应该说明什么

日常交付保持简洁：

1. Demo 文件位置。
2. 覆盖了哪些页面和状态。
3. 使用了哪些业务数据。
4. 有哪些关键假设或产品待确认项。
5. 是否存在需要设计师补充的设计缺口。
6. P0/P1/P2 自检结论。

除非产品或研发主动要求，不需要先输出长篇的组件清单、Figma 节点清单或 L0～L3 报告。

## 12. 推荐提示词

### 产品经理生成新需求 Demo

```text
读取云随项目设计 Skill 和本次需求文档，直接生成一份完整的云随 Demo。

请自行复用 Skill 中已有的业务规则、正式样板、成熟页面模式和 UDesign 映射，并补齐本次流程需要的等待、排队、Loading、结果、失败、超时、不可用、权限和历史状态。

默认交付一份可以直接双击打开的单 HTML，不要生成工程脚手架。如果存在 Skill 尚未覆盖的复杂核心页面，请列出设计缺口，不要自行创造正式方案。
```

### 研发基于现有实现改造

```text
读取云随项目设计 Skill、本次需求和现有代码，在现有实现上完成改造。

请优先复用 Skill 中已沉淀的业务规则、正式页面模式和 UDesign 映射；保留现有可用功能，不改动无关页面。完成后验证关键流程，并列出修改文件、覆盖状态、待产品确认项和设计缺口。
```

### 基于真实文件生成 Demo

```text
读取云随项目设计 Skill、本次需求和我提供的业务文件，使用文件中的真实字段和数据生成完整流程 Demo。

默认交付单 HTML，覆盖需求涉及的开始、处理中、结果和适用异常状态。Skill 中已有的设计规则和样板请自行读取；未覆盖的复杂核心设计请反馈设计缺口。
```

### 评审已有 Demo

```text
使用云随项目设计 Skill 评审这份 Demo，不要改动文件。

检查业务阶段、角色权限、页面状态、返回关系、正式样板复用和 UDesign 基础交互，并输出 P0/P1/P2 问题及修改建议。若问题来自 Skill 缺少样板或稳定规则，请单独列为设计缺口。
```

## 13. 问题反馈模板

产品或研发发现问题时，可以直接提交：

```markdown
# 云随 Demo 问题反馈

- 对应需求：
- 页面或流程：
- 当前表现：
- 期望表现：
- 是否阻断验证：是 / 否
- 是否在其他需求中重复出现：是 / 否 / 不确定
- 截图或文件：
```

设计师根据反馈判断：

- 只修改本次 Demo。
- 补充正式 Figma 样板。
- 更新业务规则或页面模式。
- 更新 UDesign 映射。
- 回灌云随 Skill。

## 14. 日常使用检查清单

### 产品或研发提交前

- [ ] 已提供云随项目设计 Skill。
- [ ] 已提供本次需求文档或业务诉求。
- [ ] 已说明要单 HTML 还是代码改造。
- [ ] 有真实业务文件时已一并提供。

### 产品或研发验收时

- [ ] 功能流程和业务内容正确。
- [ ] 角色和权限正确。
- [ ] 主操作、返回、取消和完成去向正确。
- [ ] 关键状态没有遗漏。
- [ ] 复杂核心设计缺口已明确反馈设计师。

### 设计师维护 Skill 时

- [ ] 已判断问题是单次需求还是稳定模式。
- [ ] 复杂核心页面已有正式 Figma 样板。
- [ ] 业务规则、页面模式和 UDesign 映射已同步。
- [ ] 已用真实需求回归测试。
