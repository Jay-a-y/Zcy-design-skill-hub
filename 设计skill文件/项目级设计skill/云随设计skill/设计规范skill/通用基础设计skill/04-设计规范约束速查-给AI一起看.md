# 设计规范约束基座

用途：当产品经理把 PRD、需求文档、页面说明、会议纪要交给 AI / IDE Agent 生成 Demo 或审核 Demo 时，把本文件一起给 AI。AI 必须把本文作为公司设计规范约束基座，而不是只按自己的通用审美生成页面。

本文是从本地 `乐采设计规范` HTML 资料中炼化出的 Demo 生成与审核约束，并已合并 `/Users/zcy/Desktop/AI产品设计/spec-repository-main/nova-spec/front-spec` 中的 PC 真实前端组件/API 索引。它覆盖 PC 中后台、移动端、视觉基础、组件体系、状态反馈、页面模式、数据展示、图表、文案和研发可落地规则。

注意：本文同时包含两层约束：第一层是“公司规范源页面 -> 语义组件 / 语义 Token”，用于控制页面长相、结构和体验；第二层是 `front-spec` 的“语义组件 -> 真实 PC 组件/API”，用于控制代码和研发可落地性。移动端真实代码组件名仍需后续补齐；在真实代码名不明确时，不要编造代码组件名。

## 0. 使用优先级

AI / IDE Agent 生成或审核 Demo 时，按以下优先级执行：

1. 产品经理提供的业务文档：决定业务目标、字段、状态、权限、验收口径。
2. 本设计规范约束基座：决定页面结构、组件选择、Token、状态反馈、端适配。
3. `front-spec` 前端组件/API：决定 PC 端真实组件名、props 和实现侧设计约束。
4. 通用 UX / DesignOps 方法：只在公司资料未覆盖 AI 新场景时补充，并标注为通用方法补充。

缺少业务规则时，标注 `待产品补充`；缺少设计规范时，标注 `待设计规范补充`；缺少真实前端组件/API 信息时，标注 `待前端组件库确认`。如果 `front-spec` 已有组件/API，不允许自行绕开或发明新组件。

## 2. 设计原则与产品调性

### 2.1 总体原则

- Demo 必须呈现为真实业务产品，而不是 AI 拼贴稿。
- 信息结构优先于视觉装饰。
- 主任务必须突出，主路径必须闭环。
- 页面应清晰表达业务对象、当前状态、可执行操作、异常反馈和下一步。
- 产品规则不清时，不能用 UI 包装成已确定规则。
- 设计建议和审核结论只针对 Demo，不评价产品经理个人。

### 2.2 视觉调性

- G 端 / 政务相关业务应体现方正、明晰、严谨、专业。
- 前台或面向用户的产品应体现正气、有序、进取。
- 企业业务系统整体应稳定、可信、清晰、高效。
- 不使用娱乐化、营销化、过度插画化、过度 AI 炫技的视觉表达。
- 页面应避免随机渐变、玻璃拟态、彩色发光、漂浮卡片堆叠、大面积无业务含义背景。

### 2.3 AI Demo 特别原则

- AI 能力必须服务业务任务，不应成为视觉装饰。
- AI 结果必须可解释、可确认、可编辑、可拒绝或可重试。
- AI 自动填充、AI 推荐、AI 结论不能绕过业务审核、权限和人工确认。
- AI 失败必须有手动路径、转人工路径或可恢复路径。
- AI 生成内容与用户确认后的正式内容必须区分。

## 3. Design Token 约束

### 3.1 颜色体系

公司色彩规范包含主题色、功能色、中性色、辅助色、品牌基因、层级关系和对比度要求。Demo 不能临时生成一套新主题色。

必须遵守：

- AI 生成 Demo 时必须优先从本节色板取色；如需新增颜色，必须标注 `待设计规范补充`。
- 主色只用于主操作、关键选中态、关键链接和品牌识别。
- 成功、警告、错误、信息、禁用必须使用稳定语义。
- 同一业务状态在全 Demo 中使用同一语义色。
- 危险操作必须同时具备颜色、文案和二次确认，不只依赖红色。
- 图表色必须保证可读性，不与状态色混淆。
- 大面积背景色应克制，优先保障内容层级。
- 模态遮罩使用黑色遮罩语义；移动端对话框规范中遮罩为 `#000000`、50% 不透明度。

语义 Token：

| Token | 用途 | 约束 |
| --- | --- | --- |
| `semantic.color.brand.primary` | 主操作、选中态、关键链接 | 不做大面积装饰背景 |
| `semantic.color.function.success` | 成功、通过、已完成 | 不与普通强调混用 |
| `semantic.color.function.warning` | 警告、待处理、风险提示 | 必须有明确风险说明 |
| `semantic.color.function.error` | 失败、驳回、删除、不可恢复风险 | 危险操作需二次确认 |
| `semantic.color.function.info` | 普通信息提示 | 不表达成功/失败 |
| `semantic.color.text.primary` | 标题、关键字段、关键数字 | 核心信息使用高对比 |
| `semantic.color.text.secondary` | 辅助说明 | 不用于关键字段值 |
| `semantic.color.text.disabled` | 禁用文本 | 需配合不可点击交互 |
| `semantic.color.border.default` | 表格、卡片、输入框边框 | 同类组件统一 |
| `semantic.color.background.page` | 页面背景 | PC/App 不随机换主题 |
| `semantic.color.background.container` | 容器背景 | 用于信息分组，不做漂浮装饰 |
| `semantic.color.mask.modal` | 模态遮罩 | 只用于模态阻断层 |
| `semantic.color.focus.brand` | 聚焦态 | 输入框/控件 focus 可见 |

用于图表系列、信息分类、辅助标签、科技感轻量表达。

| 层级 | 色值 |
| --- | --- |
| 深青 | `#008792`、`#009FAC` |
| 常规青 | `#00BCCC`、`#33C9D6` |
| 浅青 | `#66D7E0`、`#99E4EB`、`#CCF2F5`、`#E0F7F8`、`#F0FBFC` |

使用限制：

- 青色不能替代成功绿，除非业务语义明确。
- 图表中使用需保证与蓝色系列可区分。

##### 中性色 / 文本 / 边框 / 背景

用于文本层级、边框、分割线、禁用、背景、容器。

| 用途 | 色值 |
| --- | --- |
| 强文本 / 标题 | `#20242C`、`#232323`、`#000000` |
| 正文 / 次级文本 | `#505663`、`#565656`、`#666666` |
| 辅助文本 / 占位 | `#838B99`、`#898989`、`#A7AEBB`、`#ADADAD` |
| 边框 / 分割线 | `#C2C7D1`、`#DBDBDB`、`#E3E6EB`、`#E6E6E6`、`#ECEEF2` |

使用限制：

- 核心信息不要使用过浅灰。
- 禁用态必须同时有不可点击行为，不只是变灰。
- 表格边框、卡片边框、输入框边框要使用统一灰阶。

##### 图表推荐系列色

图表可以从以下颜色中按类别选取，避免每张图随机生成一套色板：

`#3773DA`、`#00BCCC`、`#4CB35D`、`#EF8030`、`#DA2626`、`#8060D9`、`#DD2288`、`#2EAAEE`、`#7E8DB1`、`#FAD20C`

图表使用限制：

- 同一图表中颜色数量要克制。
- 功能色不要和业务状态冲突，例如红色既表示图表分类又表示错误，会造成误读。
- 图例、坐标轴、数据标签必须可读。
- 图表标签和图形保持清晰距离，不要压住图形。

##### 禁止用色

- 不从本文色板之外随机生成品牌色。
- 不使用大面积彩虹渐变。
- 不使用彩色发光阴影。
- 不使用低对比文字。
- 不用同一颜色表达多个冲突语义。

### 3.2 字体与文本层级

公司文字规范强调可识别性、层级和阅读效率。

必须遵守：

- 最小可识别文字大小不低于 12px。
- PC 常规正文字号按 14px 处理。
- 适老化或强可读场景可放大一阶，最小 14px、常规 16px。
- 字号层级建议控制在 3-5 种，不要整页字号混乱。
- 公司资料预设 8 个基础字阶，Demo 只选择必要层级。
- 行高按字号约 1.5 倍处理，并取偶数节奏。
- 较长多行文本可在标准行高基础上增加 4px 或 8px。
- 页面标题、分组标题、字段标签、正文、辅助说明、数字指标必须区分。
- 表格、表单、列表等高密度区域优先保证扫描效率。
- 金额、数量、比例、状态数字必须对齐清楚，单位不能丢。

### 3.3 间距

间距用于表达信息关系，不用于制造装饰感。

必须遵守：

- 同组内容更近，跨组内容更远。
- PC 表格、表单、筛选区要保持紧凑但可读。
- 移动端优先使用 8n 间距节奏。
- 移动端表单模块间距可按 8n。
- 移动端弹出层内容水平间距参考 32px，垂直间距参考 24px。
- 移动单元格标题和内容之间保持稳定间距，页面内标题宽度应统一。
- 图表数据标签与图形保持可读距离，资料中出现数据标签与图形保持 4px 的规则。
- 不使用随机大留白、过度宽松卡片或营销式空间。

### 3.4 圆角

必须遵守：

- 同类组件圆角一致。
- 常用圆角层级可按 0 / 2 / 4 / 8 / 12 / 16 / 20px 理解。
- PC 高密度表格、表单、后台容器不使用过大圆角。
- 标签、输入框、按钮、卡片、弹窗、浮层应使用各自稳定圆角。
- 移动端弹出层、卡片、按钮跟随公司 App 圆角体系，不自造超大圆角 AI 风格。
- 圆角不能替代信息层级。

### 3.5 边框与分割线

必须遵守：

- 常规边框以 1px 为主。
- 如无必要，边框粗细尽量不超过 2px；最大不超过 4px。
- 常规边框使用实线。
- 虚线边框仅用于上传、拖拽、占位等明确场景，虚线节奏不随意调整。
- 白色背景上的卡片、表格边框使用稳定浅灰边框语义。
- 分割线用于区分信息层级，不能替代模块间距。
- 表格线、列表线、弹窗分割线、表单组分割线应统一灰阶和粗细。

### 3.6 阴影与质感

必须遵守：

- 阴影只用于表达层级和交互，不用于装饰。
- 基础阴影为带透明度黑色系，不使用彩色发光阴影。
- 输入框聚焦态可使用 `0 0 0 2px` 的品牌聚焦阴影语义。
- 小阴影：卡片 hover、轻微突出元素。
- 中阴影：下拉、气泡、非模态浮层。
- 大阴影：弹窗、抽屉、最高层级模态信息面板。
- 如果组件默认无阴影，但业务确需阴影，应标注 `待设计规范/前端组件库确认`，不要自行改组件。

### 3.7 栅格、布局与层级

PC：

- 框架由背景层、内容层、导航层、事件层组成。
- 层级顺序必须保持：`事件层 > 导航层 > 内容层 > 背景层`。
- 事件层包括弹窗、浮层、提示、分支任务。
- 导航层包括顶部导航、左侧菜单、面包屑等。
- 内容层承载表格、表单、详情、看板。
- 页面布局遵循预知性、一致性、响应性。
- 栅格宽度随设备宽度变化，高度由内容决定。

移动端：

- 依据移动端设计尺寸、栅格和布局规范组织页面。
- 一屏一个核心任务。
- 不把 PC 多列布局直接缩小。
- 底部主操作不能遮挡内容、错误提示或键盘输入。

### 3.8 图标、图片、插图

必须遵守：

- 图标必须服务操作或状态。
- 同一页面图标线性/面性风格、尺寸、描边粗细保持一致。
- 不使用无语义装饰图标填充页面。
- 空状态、缺省页、结果页可使用插图，但必须解释发生了什么和用户能做什么。
- 图片预览、图片上传、附件上传必须有加载、失败、删除、重试状态。
- 图标不能替代必要文字说明，尤其是高风险操作。

### 3.8.1 PC 按钮与图标对齐规范

本节是 PC Demo 的硬约束。按钮图标如果无法映射到公司组件库图标、统一 SVG 图标或前端组件库 Icon，应优先去掉图标，只保留清晰文字，不允许用临时符号凑视觉效果。

图标来源：

- 必须使用公司组件库图标、统一 SVG 图标或前端组件库 Icon。
- 不允许使用 Unicode 字符、emoji、特殊符号临时代替图标，例如 `⌕`、`↺`、`□`、`✓`、`×`、`↓`、`‹`、`›`。
- 不确定真实图标名称时，输出语义图标名，例如 `icon.search`、`icon.reset`、`icon.download`，并标注 `待前端组件库确认`，不要自行发明图标样式。

PC 文本按钮默认不强制带图标。以下基础操作优先只用文字：

- 查询
- 重置
- 查看
- 展开更多 / 收起更多
- 清空选择
- 上一页 / 下一页
- 取消 / 返回

适合带图标的场景：

- 导入、导出、上传、下载。
- 新建、删除、批量处理等需要提高识别度的操作。
- 纯图标按钮，例如刷新、关闭、折叠，仅在图标语义明确且有 `title` / `aria-label` 时使用。
- 高风险操作不只依赖图标，必须有明确文字。

按钮内部布局：

- 按钮必须使用 `inline-flex` 或公司按钮组件默认布局。
- 必须 `align-items: center`、`justify-content: center`。
- 图标和文字间距固定为 4px 或 6px，同一页面保持一致。
- 图标不得单独改变按钮高度。
- 图标不得上浮、下沉、偏左、偏右。
- 图标不得比文字更抢眼。

PC 常规按钮建议：

| 项 | 建议 |
| --- | --- |
| 默认按钮高度 | 32px |
| 小按钮高度 | 28px |
| 强操作按钮高度 | 32px 或 36px，同一区域保持一致 |
| 文本字号 | 14px |
| 图标尺寸 | 14px 或 16px |
| 图标和文字间距 | 4px 或 6px |
| 图标风格 | 同一页面保持线性/面性、描边粗细一致 |

按钮禁止项：

- 不使用 Unicode 符号当图标。
- 不在同一操作区混用不同大小图标。
- 不让图标独占过大宽度。
- 不让图标与文字基线错位。
- 不为了“看起来更丰富”给所有按钮都加图标。
- 不把查询、重置、查看、展开更多等基础操作做成视觉很重的图标按钮。
- 不在禁用按钮中保留高对比图标，禁用态文字和图标都应同步弱化。

### 3.9 动效

必须遵守：

- 动效只用于状态变化、层级切换、反馈和引导。
- 动效节奏应轻量，不影响业务效率。
- 不用动效掩盖状态缺失。
- AI 生成中可以有动效，但必须有文字说明当前处理内容。

## 4. PC Demo 信息密度与复杂度治理

本章优先级高于具体组件展示规则。AI 生成 PC Demo 时，不是把所有规范、所有状态、所有服务能力都展示出来，而是按主任务选择性呈现。Demo 要像真实业务系统，不像规范验收清单。

### 4.1 UI 页面与说明材料必须分离

必须遵守：

- UI 页面只展示用户完成主任务所需内容。
- 需求摘要、设计规范执行摘要、Component / Token Usage Manifest、状态覆盖清单、待补规则清单，不得作为业务页面区块展示。
- Manifest 必须作为 Demo 生成后的文本输出，不进入可视 UI。
- 规范覆盖情况可以写在生成结果说明里，不能放进页面主体、抽屉、弹窗或卡片。
- 页面中不出现“以下内容按 IDE-Agent 输入文档输出”“公司组件映射”“语义 Token”“待产品补充规则”等交付说明型区块。

允许：

- 页面中出现必要业务提示，例如“当前筛选条件下无结果”“提交后不可撤回”。
- 开发/评审说明作为代码注释或生成后文本说明输出。

### 4.2 第一屏四要素

PC Demo 第一屏只优先回答 4 件事：

1. 当前对象：用户正在处理哪个项目、标项、订单、供应商、文件或业务对象。
2. 当前状态：该对象处于什么业务状态。
3. 主任务：用户当前最应该完成什么。
4. 主操作：用户下一步最重要的操作是什么。

第一屏禁止：

- 同时展示多个互不相关业务模块。
- 同时展开全部状态、全部服务、全部规则说明。
- 用大段说明文案挤占主任务区域。
- 为了证明规范完整而展示 Manifest、状态覆盖、组件映射。

### 4.3 页面区块数量控制

默认区块上限：

| 页面类型 | 推荐区块数 | 区块示例 |
| --- | --- | --- |
| 列表页 | 3-5 个 | 页头、筛选、表格、分页/状态、必要辅助信息 |
| 表单页 | 3-5 个 | 页头、基础信息、业务字段组、附件/补充信息、操作区 |
| 详情页 | 4-6 个 | 摘要、基础信息、业务明细、附件、流转日志、相关对象 |
| 看板页 | 4-6 个 | 筛选、指标卡、核心图表、异常列表、口径说明 |
| AI 工作台 | 3-5 个 | 服务入口、当前服务摘要、问题/结果列表、操作区、依据抽屉入口 |

超过上限时必须收敛：

- 合并同类信息。
- 次要信息折叠。
- 详情进入抽屉或二级页。
- 非当前任务服务只保留入口。

### 4.4 按钮、标签、说明文案上限

按钮：

- 每个操作区只保留 1 个主按钮。
- 次按钮不超过 2 个。
- 更多操作进入“更多”或下拉菜单。
- 表格行内操作建议 2-3 个以内，低频操作收起。
- 按钮是否带图标由业务识别需要决定，基础操作优先文字按钮。
- 同一操作区按钮高度、图标尺寸、图标和文字间距必须一致。
- 查询、重置、查看、展开更多、清空选择等操作不要为了装饰强行加图标。

标签：

- 单个对象默认展示 1-3 个标签。
- 标签只表达状态、风险、分类或重要属性。
- 不为每个字段都加标签。
- 多个标签同时出现时，必须有优先级。

说明文案：

- 区块说明不超过 2 行。
- 长规则说明进入 Tooltip、Popover、抽屉或帮助入口。
- 不把 PRD 段落直接搬到页面上。
- 页面文案服务操作判断，不服务内部交付说明。

### 4.5 状态展示收敛

必须设计状态，但不允许全量常驻平铺。

要求：

- 加载、空、错误、无权限、禁用、提交中、提交失败是状态设计，不是默认同时展示的内容。
- 默认页面只展示当前真实状态。
- 其他状态在生成后说明中列出，或通过交互/切换模拟。
- 如果需要演示多状态，用“状态切换控件”或独立状态预览，不和主业务页面混在一起。

禁止：

- 在表格标题旁同时写“查询成功 / 加载中 / 无结果 / 接口失败 / 无权限”。
- 在同一业务区块里同时展示空态、错误态、加载态。
- 为了证明覆盖状态而压缩真实业务内容。

### 4.6 信息优先级

信息分为 4 层：

1. P0 核心任务信息：对象、状态、主操作、关键字段。
2. P1 决策辅助信息：风险、金额、时间、责任人、异常原因。
3. P2 追溯信息：日志、附件、历史记录、依据。
4. P3 交付说明信息：需求摘要、规范执行摘要、Manifest、待补规则。

展示规则：

- P0 必须在主视区可见。
- P1 可以在主视区或紧邻区域展示。
- P2 放到详情、抽屉、折叠面板。
- P3 不进入 UI，只作为生成后文本输出。

## 5. PC 页面间距与布局收敛规则

### 5.1 区域级间距

推荐节奏：

| 层级 | 建议间距 |
| --- | --- |
| 页面外层左右留白 | 16px / 20px / 24px，按页面密度选择 |
| 页头与内容区 | 16px - 24px |
| 区块之间 | 16px - 24px |
| 区块标题与内容 | 8px - 12px |
| 表单行之间 | 12px - 16px |
| 表格工具栏与表格 | 8px - 12px |
| 按钮组间距 | 8px |
| 标签组间距 | 4px - 8px |

要求：

- 同一页面只使用一套稳定间距节奏。
- 区块间距大于区块内间距。
- 表格、表单、筛选区保持紧凑但可读。
- 不用大留白制造高级感。

### 5.2 卡片与容器收敛

禁止：

- 卡片套卡片。
- 每个模块都独立加边框和阴影。
- 区块过度包边，导致页面碎片化。
- 为了区分层级而堆叠大量浅色容器。

推荐：

- PC 中后台优先使用表格、表单、详情分组。
- 信息卡片只用于摘要、指标、对象概览。
- 白底容器 + 分割线优先于多层卡片堆叠。
- 阴影只用于浮层、抽屉、弹窗、hover，不用于普通区块装饰。

### 5.3 筛选区收敛

要求：

- 筛选区默认一行优先。
- 1-4 个筛选条件可直接展示。
- 超过 4 个筛选条件时使用展开/收起。
- 低频筛选进入“更多筛选”。
- 查询和重置按钮位置稳定。
- 筛选区说明文案不常驻展示，必要时放 Tooltip。

禁止：

- 把全部筛选条件铺成多行大表单。
- 筛选区说明比筛选控件更醒目。
- 每个筛选字段都配长说明。

### 5.4 表格和详情布局收敛

表格：

- 优先展示最影响判断的列。
- 次要字段进入详情抽屉。
- 行内操作收敛到主操作 + 更多。
- 长文本截断，详情通过 Tooltip 或抽屉查看。

详情：

- 摘要区只展示状态和关键字段。
- 基础信息按 2-3 列组织。
- 业务明细和日志分区展示。
- 附件、日志、依据默认可折叠或放后置区块。

## 6. AI 工作台 / 右侧抽屉类 Demo 约束

适用于“宿主业务系统 + AI 悬浮入口 / AI 右侧工作台 / AI 抽屉”的 Demo。

### 6.1 宿主和 AI 工作台分层

宿主业务页面：

- 只展示业务主流程。
- 只保留 AI 入口、关键 AI 状态或主操作。
- 不展开所有 AI 服务能力。

AI 工作台：

- 承载 AI 服务列表、当前服务详情、检测/推荐/提取结果。
- 默认只展开当前服务。
- 非当前服务只保留入口、可用状态和简短说明。

### 6.2 AI 服务展示收敛

禁止：

- 同时展开采购文件提取、检测、推荐、围串标识别、辅助评标等全部服务详情。
- 账户信息、授权弹窗、服务矩阵、检测结果、推荐结果同时占用主工作区。
- 为展示能力而把所有阶段、角色、服务、状态全部铺开。

推荐：

- 首页只展示服务入口和当前可用性。
- 当前服务页展示摘要、问题列表、主操作。
- 详细依据进入抽屉、Popover 或二级页。
- 账户、授权、设置放 Popover/Dialog，不常驻主区。
- 服务矩阵作为辅助说明，不作为默认主内容。

### 6.3 AI 检测/推荐结果布局

AI 检测结果优先结构：

- 检测摘要。
- 风险数量。
- P0/P1/P2 或高/中/低风险列表。
- 每条问题的定位入口。
- 查看依据入口。
- 修复建议。

AI 推荐结果优先结构：

- 当前推荐章节/对象。
- 推荐来源：机构历史库、平台脱敏库、AI 生成。
- 推荐理由。
- 可引用条款列表。
- 一键引用或选择引用。
- 替代方案入口。

禁止：

- 把完整原文、完整推荐、完整检测细则全部铺在同一屏。
- AI 结果没有人工确认就直接回填。
- AI 结果和正式业务状态混淆。

- Hover 状态：hover 滑块时变为可点击状态；hover 数值条时显示当前位置数值提示。
- Active 状态：按住滑块进入选中态。
- 拖动联动：拖动时数值随位置变化，输入框数值同步变化。
- 输入联动：输入框数值变化后，滑块位置同步变化。
- 边界规则：小于最小值置为最小值，大于最大值置为最大值。
- 点击轨道：点击数值条时滑块移动到点击位置。
- 范围滑动：两个滑块可在数值条上拖动，保证第一个值不大于第二个值。
- 分步滑动：只能移动到刻度值；点击标签定位到对应刻度。
- 禁用状态：鼠标不可点击，必要时仍可显示当前位置数值。

- S：内部列表或瀑布流加载，参考 32px、描边 4px。
- M：卡片或局部模块加载，参考 48px、描边 6px。
- L：页面级中心加载，参考 64px、描边 8px。
- 可配合文字说明，说明当前处理内容。

空状态：

- 说明发生了什么。
- 说明用户能做什么。
- 必要时提供重置筛选、新建、返回工作台等操作。

结果页：

- 必须说明提交/处理结果。
- 展示编号、预计时间、当前节点、下一步。
- 主操作只有一个。
- AI 建议不能改变正式审核状态，除非业务规则明确允许。

## 10. 图表与数据可视化

来源：图表组成、图表类型、坐标轴、图例、数据标签、折线类、柱状类、条形类、饼环类、面积类。

要求：

- 图表必须服务业务判断，不做装饰。
- 必须说明指标口径、统计周期、单位和数据范围。
- 坐标轴刻度、单位、范围清楚。
- 图例与图表颜色对应。
- 数据标签可读，资料中出现 12 号字、中等字重、墨灰色、与图形保持 4px 距离的规则。
- 折线图用于趋势。
- 柱状图和条形图用于分类对比。
- 饼图/环图用于占比，不用于过多分类。
- 面积图用于趋势和规模感。
- 异常、峰值、低值应有说明。
- 图表下钻路径或明细入口要清楚。

## 11. 金额、标签、文案

### 11.1 金额

来源：金额展示、金额输入。

要求：

- 金额必须带单位和币种口径。
- 金额数字对齐。
- 大金额展示应考虑千分位、精度、单位转换。
- 金额输入需有格式校验、最大/最小值、精度、错误提示。
- 金额相关风险操作必须有确认。

### 11.2 标签

来源：中后台、移动端、前台标签规范。

要求：

- 标签用于状态、属性、分类、风险、服务承诺等。
- 标签数量克制。
- 同级标签视觉强弱一致。
- 高优先级标签先展示。
- 移动端标签需考虑一行内容容量。
- 价格、优惠、服务、政策类标签不得互相抢层级。

### 11.3 文案

来源：文案规范、提示信息、协议通知类规范。

要求：

- 按用户语言表达，不写内部黑话。
- 按操作结果写文案，不写空泛提示。
- 按钮文案必须具体，如“提交审核”“保存草稿”“确认下架”。
- 错误文案说明原因和下一步。
- 空状态文案说明发生了什么和能做什么。
- 高风险操作文案说明影响范围。
- 协议、通知、政策透传类信息必须清楚说明来源、影响和用户动作。

## 12. 空状态、缺省页、错误页

要求：

- 空状态不能只有插图。
- 必须说明为什么为空。
- 必须说明下一步可做什么。
- 筛选无结果应提供重置筛选。
- 首次无数据可提供新建入口。
- 无权限应说明权限限制和联系/申请路径。
- 接口错误应提供重试。
- 移动弱网应保留用户已输入内容。

## 13. AI 场景设计约束

公司资料未完整覆盖 AI 新场景，以下为通用方法补充，但必须落在公司组件体系内。

### 13.1 AI 对话

- 使用公司列表、卡片、输入框、Loading、Toast、结果页等基础组件组合。
- 区分用户输入、AI 输出、系统提示。
- AI 回复结构化为：结论、依据、建议、下一步。
- 长回答折叠为要点。
- 提供复制、查看依据、重新生成、转人工。
- 不使用独立 AI 皮肤破坏公司组件体系。

### 13.2 AI 结果确认

- 展示输入摘要。
- 展示生成结果。
- 展示依据或来源。
- 支持确认采用、编辑、拒绝、重新生成。
- 高风险结果必须人工确认。
- 采用后状态要明确。

### 13.3 AI 推荐

- 说明推荐理由。
- 说明适用条件。
- 说明风险提示。
- 提供替代方案。
- 不把推荐包装成唯一正确答案。

### 13.4 AI 填单

- AI 填充字段必须可核对、可修改。
- 低置信字段必须标记。
- 敏感字段提交前必须人工确认。
- AI 失败时提供手动填写或转人工路径。
- AI 生成内容与用户确认后的正式提交内容必须区分。

## 14. 组件状态完整性

所有 Demo 必须根据场景覆盖相关状态。

通用状态：

- `state.normal`
- `state.hover`
- `state.focus`
- `state.active`
- `state.disabled`
- `state.loading`
- `state.success`
- `state.error`
- `state.empty`
- `state.warning`
- `state.noPermission`

PC 重点：

- 表格：加载、空、错误、选中、批量选择、行 hover、排序、筛选、分页、长文本溢出。
- 表单：默认、聚焦、必填、错误、禁用、只读、保存中、保存成功、保存失败。
- 弹窗：打开、关闭、提交中、提交失败、二次确认、取消确认。
- 抽屉：打开来源、保存、关闭、返回上级、叠加层级。
- 按钮：主按钮、次按钮、危险按钮、禁用、加载中。

移动端重点：

- 列表：首屏加载、下拉刷新、上拉加载、空、失败、弱网重试。
- 表单：输入中、错误、键盘遮挡处理、草稿保存、提交中、提交失败。
- 底部操作区：默认、禁用、加载中、键盘弹起后的避让。
- 弹出层：打开、滚动、确认、取消、键盘避让。
- 对话框：警告、确认、单按钮、多按钮、不可点遮罩关闭。

## 15. 研发可落地约束

Demo 必须让研发能理解：

- 页面结构。
- 组件拆分。
- 字段类型。
- 字段必填/只读/禁用。
- 状态机。

## 17. 评审分级

P0 阻断型问题：

- 影响核心流程。
- 影响业务规则。
- 影响用户完成主任务。
- 影响研发理解。
- 影响验收通过。
- 产品规则缺失。
- 关键状态缺失。
- 移动端主链路不可用。
- 组件/Token 错误导致核心流程不可完成或研发无法理解。
- PC 真实组件/API 使用错误，导致研发无法按公司组件库落地。

P1 体验效率问题：

- 不阻断主流程，但影响用户理解。
- 信息层级不清。
- 操作路径偏长。
- 组件不统一。
- 页面结构影响效率。
- 页面信息过载、区块过多、按钮/标签过多。
- Manifest、规范清单、交付说明进入业务 UI。
- AI 工作台和宿主业务页面层级混乱。
- 移动端适配不佳。
- 大量未映射组件或随机 Token 影响理解、效率或研发复用。
- 代码绕开 `front-spec` 已有组件，大量手写控件或自造 props。

P2 体验细节问题：

- 颜色、间距、图标、微文案、局部一致性。
- 局部间距、标签密度、按钮组间距、卡片密度问题。
- 局部 Token 或组件状态不一致，但不影响主流程和研发理解。

## 17.1 失败样本分级与规则回收

AI Demo 生成失败样本按 F0/F1/F2/F3 记录到 `evidence/`，用于后续回收规则。

| 分级 | 定义 | 处理方式 |
| --- | --- | --- |
| F0 | 完全跑偏，例如生成营销页、端类型错误、主任务不存在 | 阻断，必须重生成；回收到生成前规则 |
| F1 | 主流程可见但产品化风险高，例如信息过载、组件混乱、Manifest 入侵 UI、front-spec 已有组件却手写控件 | 本期修正；回收到 `quick-use/04`、`01A/02A/03A` 或组件映射 |
| F2 | 局部体验问题，例如间距、按钮图标、文案、标签、圆角局部不一致 | 可后续优化；视频率决定是否回收 |
| F3 | 规范未覆盖的新场景，不一定是生成错误 | 进入 evidence，人工判断是否扩展规范 |

规则回收流程：

1. 真实 Demo 出现问题。
2. 审核报告标注 P0/P1/P2，同时将生成失败归类为 F0/F1/F2/F3。
3. 高频或高风险问题进入 `evidence/`。
4. 判断回收位置：`quick-use/01A`、`02A`、`03A`、`04`、`shared/component-token-mapping/`。
5. 更新规则后，用 `validation/` 回归用例验证。

注意：单一项目的特殊业务规则不直接写入通用规范，先保留在项目文档或 evidence 中观察。

## 18. AI 使用本文时的执行要求

生成 Demo 前：

1. 先读取产品文档。
2. 抽取业务目标、角色、页面、字段、状态、权限、验收标准。
3. 按本文选择 PC 或移动端组件。
4. 如果是 PC 端代码生成，按 `front-spec/INDEX.md` 读取真实组件/API，并映射到 `Button`、`Table`、`Form`、`Input`、`Select`、`Modal` 等真实组件。
5. 按本文选择语义 Token 和状态。
6. 缺失规则标注 `待产品补充`。
7. 将 UI Demo 和说明材料分离输出。
8. 输出 Demo 和 Component / Token Usage Manifest；Manifest 不进入 UI。

审核 Demo 时：

1. 先读取 Demo、产品文档、验收标准。
2. 判断是否符合本文的组件、Token、状态和端适配约束。
3. 如果是 PC 端 Demo，检查是否符合 `front-spec` 真实组件/API。
4. 输出 P0/P1/P2。
5. 区分视觉问题、交互问题、产品规则问题、研发落地问题。
6. 给出准入结论。
7. 明确 output-report 不等于人工验收。
8. 对高频问题给出 evidence 和规则回收建议。
