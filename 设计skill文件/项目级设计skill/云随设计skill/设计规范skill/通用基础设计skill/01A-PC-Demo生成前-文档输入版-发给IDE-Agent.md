使用方式：产品经理不用逐项填空，只需要把自己的需求文档、PRD、页面说明或会议纪要和本文件一起发给 IDE Agent。

目标：让 IDE Agent 先从产品文档中自动抽取业务信息，再按 `04-设计规范约束速查-给AI一起看.md` 中的完整公司设计规范、PC 组件映射、Design Token 和 `front-spec` 真实组件/API 生成 Demo。

### 2. 页面生成原则

- 生成 PC 中后台 Demo，不要生成营销页。
- 页面第一屏必须能看出当前模块、主对象、主任务、主操作和当前状态。
- 使用 PC 中后台结构：导航、页面标题、筛选区、内容主体、操作区、反馈区。
- PC 适合高密度信息处理，表格、表单、详情、看板要清晰高效。
- 不要为了“AI 感”增加随机渐变、玻璃拟态、发光背景、大面积插画。
- 所有视觉、组件、状态、图表、文案和空状态规则，必须按 `04-设计规范约束速查-给AI一起看.md` 执行。
- 必须遵守 `04` 中的“PC Demo 信息密度与复杂度治理”：不要把所有规范、状态、服务能力一次性铺开。

### 3. 生成前信息收敛流程

生成 UI 前必须先做收敛，不要直接把产品文档里的所有内容铺成页面。

1. 从产品文档抽取需求摘要。
2. 判断本次 Demo 的主链路和主任务。
3. 判断页面模式：列表、详情、表单、看板、AI 工作台、弹窗/抽屉。
4. 选择当前页面必须展示的 P0/P1 信息。
5. 将日志、依据、附件、规则说明等 P2 信息放入详情、抽屉、折叠面板或生成后说明。
6. 将需求摘要、设计规范执行摘要、Component / Token Usage Manifest、状态覆盖清单放到生成后的文本说明，不进入 UI。
7. 如果产品文档内容很多，优先生成主链路；不要把所有章节、状态、服务、规则同时放进一个页面。

UI 页面必须遵守：

- 每个页面只保留一个主任务。
- 每个操作区只保留一个主按钮。
- 第一屏只回答当前对象、当前状态、主任务、主操作。
- 列表页默认 3-5 个区块；看板页默认 4-6 个区块；详情页按摘要、基础、明细、流转组织。
- 标签默认 1-3 个；辅助说明不超过 2 行。

- 表单：`pc.input.Form`、`pc.input.TextInput`、`pc.input.Select`、`pc.input.DatePicker`、`pc.input.TimePicker`、`pc.input.FileUpload`、`pc.input.ImageUpload`、`pc.input.Cascader`、`pc.input.TreeSelect`
- 展示：`pc.display.DescriptionList`、`pc.display.InfoCard`、`pc.display.Timeline`、`pc.display.Tag`
- 反馈：`pc.feedback.Toast`、`pc.feedback.Alert`、`pc.feedback.Loading`、`pc.feedback.Dialog`、`pc.feedback.Drawer`

### 4.1 真实前端组件/API 生成约束

PC Demo 如果生成真实代码，必须优先映射到 `front-spec` 中已有组件：

| Demo 语义 | 真实组件/API | 生成要求 |
| --- | --- | --- |
| 按钮 | `Button`、`ButtonConfirm`、`api/Button.md` | 使用真实 `type/icon/size/loading/disabled/danger/htmlType` 等属性；不要自造 `variant` 或 Unicode 图标 |
| 表格 | `Table`、`Table.Actions`、`Tabulation`、`api/Table.md` | 行操作用 `Table.Actions` 或等价公司组件能力；不要每行堆多个大按钮 |
| 表单 | `Form`、`Form.Item`、`ZcyFormItem`、`FormGrid`、`ZcyGrid` | 必填、校验、错误、禁用、只读和分组规则完整 |
| 输入 | `Input`、`Input.Search`、`ZcySearch` | 搜索和普通输入分开；支持清空、字数、禁用和错误态 |
| 选择 | `Select`、`Select.Option`、`DatePicker`、`TimePicker` | 明确选项、空值、范围、禁用和弹层容器规则 |
| 页头/路径 | `ZcyPageHeader`、`ZcyBreadcrumb` | 表达标题、状态、摘要、主操作和返回路径 |
| 弹窗/确认 | `Modal`、`Modal.confirm`、`Popconfirm` | 标题、内容、主次按钮、关闭路径和危险操作二次确认完整 |
| 加载/提示 | `ZcySpin`、`Spin`、`Skeleton`、`message`、`Alert` | 区分按钮级、局部区域、整页加载和页面内提示 |

生成前检查：

- 如果已有真实组件，不要手写 `div/button/table` 模拟。
- 如果需要的组件不在 `front-spec` 中，标注 `待前端组件库确认`，并说明临时替代方案。
- 如果只生成静态 HTML，也要在生成后文本中列出对应真实组件/API，不要放进 UI。

### 5. 按钮与图标生成要求

生成 PC Demo 时，按钮必须优先使用公司按钮组件语义：

- `pc.input.Button.primary`
- `pc.input.Button.secondary`
- `pc.input.Button.danger`
- `pc.input.Button.text`
- `pc.input.Button.iconOnly`

按钮图标硬约束：

- 不要用 Unicode 字符、emoji 或特殊符号模拟图标，例如 `⌕`、`↺`、`□`、`✓`、`×`、`↓`、`‹`、`›`。
- 如果没有明确的公司图标或前端图标组件，优先去掉图标，只保留文字。
- 如果必须表达图标语义，使用语义图标名说明，例如 `icon.search`、`icon.reset`、`icon.download`，并在生成后说明中标注 `待前端组件库确认`。
- 同一页面内按钮图标尺寸只允许使用一套，建议 14px 或 16px。
- 同一页面内按钮图标风格、描边粗细、颜色和垂直对齐必须一致。
- 图标和文字间距固定为 4px 或 6px，不允许每个按钮不同。
- 图标不得导致按钮高度变化、文字偏移或视觉重心不居中。

默认规则：

- 查询、重置、查看、展开更多、收起更多、清空选择、上一页、下一页等基础操作可以只用文字按钮。
- 导入、导出、上传、下载、新建、删除、批量处理等识别强的操作可以使用图标。
- 纯图标按钮必须有 `title` / `aria-label`，并且只用于刷新、关闭、折叠、设置等语义明确场景。
- 高风险操作必须有文字，不允许只用图标表达。

CSS / 组件实现要求：

- 按钮使用公司按钮组件默认布局，或使用 `inline-flex`。
- 必须保证 `align-items: center`、`justify-content: center`。
- 按钮文字字号默认 14px。
- PC 常规按钮高度默认 32px，小按钮 28px，强操作按钮可为 32px 或 36px；同一区域按钮高度必须一致。
- 禁用按钮的文字和图标都要同步弱化，不允许图标仍然高亮。

输出前自检：

- 每个按钮图标是否来自同一图标体系？
- 是否仍有 Unicode 字符或特殊符号充当图标？
- 图标尺寸是否统一？
- 图标和文字是否垂直居中？
- 图标和文字间距是否统一？
- 查询、重置、查看、展开更多等基础按钮是否可以去掉图标？
- 表格行操作按钮是否过重、过大或图标错位？

### 6. 必须遵循的 Design Token 约束

必须遵守 `04` 中完整的 Design Token 约束。以下是最低要求：

- 主色只用于主操作、关键选中态、关键链接。
- 成功、警告、错误、禁用状态必须使用稳定语义。
- PC 常规正文按 14px 层级处理，最小可识别文字不低于 12px。
- 字号层级控制在 3-5 种。
- 同组内容更近，跨组内容更远。
- 同类组件圆角一致，不混用多套圆角。
- 常规边框以 1px 为主。
- 阴影只用于浮层、抽屉、弹窗、卡片 hover 等层级表达。
- 不使用彩色发光阴影。

### 7. 页面类型规则

列表页必须包含：

- 搜索/筛选
- 重置
- 全局操作
- 批量操作
- 表格
- 分页
- 空状态
- 加载状态
- 错误状态

表单页必须包含：

- 字段分组
- 必填标识
- 字段说明
- 字段校验
- 错误提示
- 保存/取消/提交
- 提交中/成功/失败反馈

详情页必须包含：

- 摘要
- 状态
- 主操作
- 基础信息
- 业务明细
- 附件
- 流转日志

弹窗/抽屉必须包含：

- 标题
- 内容
- 主次按钮
- 关闭路径
- 操作反馈
- 来源页或返回路径

### 8. 状态要求

必须覆盖与业务相关的状态：

- 查询：加载中、查询成功、无结果、接口失败。
- 表格：空值、长文本、排序、筛选、分页、批量选择、行操作。
- 表单：默认、聚焦、必填、错误、禁用、只读、保存中、保存成功、保存失败。
- 删除/批量：二次确认、处理中、成功、部分失败、无权限。
- 审批/提交：提交前确认、提交中、提交成功、驳回/失败原因、查看进度。
- AI 相关模块：生成中、生成失败、结果可编辑、人工确认、重新生成。

状态呈现要求：

- 状态必须设计完整，但 UI 默认只展示当前真实状态。
- 加载、空、错误、无权限、禁用等状态不得常驻平铺在主页面。
- 如需演示多状态，请用状态切换或生成后说明，不要挤占业务页面。

### 10. 反冗余禁止事项

- 不要生成营销页。
- 不要自造公司规范之外的新组件。
- 不要绕开 `front-spec` 已有组件，大量手写控件或自造 props。
- 不要把 `pc.input.Button.primary`、`pc.display.Table` 等 Skill 语义名当真实代码组件名。
- 不要让 AI 装饰效果压过业务可读性。
- 不要用 Unicode 字符、emoji 或特殊符号冒充按钮图标。
- 不要为了“看起来更完整”给所有按钮都加图标。
- 不要让按钮图标大小、位置、描边、颜色在同一页面内不一致。
- 不要把移动端卡片当成 PC 表格主方案。
- 不要把产品规则缺失伪装成视觉问题。
- 不要把规范清单、Manifest、需求摘要做成页面区块。
- 不要把所有状态常驻展示在页面中。
- 不要把所有 AI 服务同时展开。
- 不要把所有业务规则写成长段说明。
- 不要为证明“覆盖完整”而增加无主任务区块。
- 不要让按钮、标签、卡片数量超过当前主任务需要。
