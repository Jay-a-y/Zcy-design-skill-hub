# Demo 生成后 AI 审核提示词：文档输入版

使用方式：产品经理不用逐项填空，只需要把 Demo 链接/截图/录屏、产品文档、验收标准和本文件一起发给 IDE Agent。

目标：让 IDE Agent 先自动读取材料，再按 `04-设计规范约束速查-给AI一起看.md` 中的完整公司设计规范、组件映射、Design Token 和产品化标准审核 Demo。

如果审核对象是 PC 端代码或声明了真实组件，还必须读取 `/Users/zcy/Desktop/AI产品设计/spec-repository-main/nova-spec/front-spec/INDEX.md` 和实际用到组件对应的 `api/*.md`，检查是否符合公司真实组件/API，而不是只看视觉像不像。

### 2. 审核维度

请逐项审核：

1. 业务目标是否明确。
2. 核心用户路径是否闭环。
3. 页面主任务是否突出。
4. 信息层级是否清晰。
5. 字段、规则、状态是否完整。
6. PC / 移动端适配是否合理。
7. 是否存在 PC 方案直接套移动端。
8. 交互反馈、异常状态、空状态、加载状态是否完整。
9. 视觉风格是否符合业务系统稳定、可信、清晰、高效。
10. 组件、颜色、字体、间距、圆角、图标是否统一。
11. 是否符合公司组件库 / Design Token 映射。
12. 是否具备研发可落地性。
13. 是否存在“看起来像 AI 生成，但不是真实产品”的问题。
14. 是否存在产品规则缺失却要求设计兜底的问题。
15. 是否把交付说明、Manifest、规范清单、需求摘要展示进业务页面。
16. 第一屏是否能清楚回答当前对象、当前状态、主任务、主操作。
17. 页面是否区块过多、标签过多、按钮过多、说明文案过多。
18. 筛选区、表格区、详情区、抽屉区是否有合理间距。
19. AI 工作台是否和宿主业务页面层级混乱。
20. 是否存在“为了展示能力而把所有功能摊开”的问题。
21. 按钮图标是否来自统一图标体系，而不是 Unicode 字符、emoji 或特殊符号。
22. 按钮图标尺寸、垂直对齐、描边粗细、颜色、图标与文字间距是否一致。
23. 查询、重置、查看、展开更多等基础按钮是否存在不必要图标，导致页面显得杂乱。
24. PC 端真实代码是否使用 `front-spec` 已有组件，而不是大量手写控件。
25. 组件 props 是否能在 `front-spec/api/` 找到依据，是否存在自造 props。
26. 表格行操作是否使用 `Table.Actions` 或等价公司组件能力，避免每行大按钮堆叠。
27. Manifest 是否同时列出 Skill 语义组件、front-spec 真实组件/API 和待确认项。
28. 是否把 AI 审核报告误当成人工验收结论。

### 3. 公司组件/Token 审核标准

请按 `04` 的完整规则检查，以下是最低检查项：

- PC 页面是否能映射到公司 PC 组件：表格、表单、弹窗、抽屉、详情、导航、反馈。
- 移动端页面是否能映射到公司移动端组件：卡片、列表、单元格、表单、弹出层、对话框、加载、结果页。
- 颜色、字号、间距、圆角、边框、阴影是否使用稳定语义 Token。
- 是否存在未映射 AI 风格控件，例如随机智能卡片、发光输入框、玻璃拟态浮层、过度装饰背景。
- 未映射组件是否影响主流程、研发复用或验收判断。
- 按钮是否能映射到公司按钮组件语义：主按钮、次按钮、危险按钮、文字按钮、纯图标按钮。
- 按钮图标是否能映射到公司图标库或统一 SVG 图标；无法映射时是否已去掉图标或标注 `待前端组件库确认`。
- PC 代码组件是否能映射到 `front-spec-real-component-map.md` 和 `front-spec/api/*.md`。

### 3.2 front-spec 真实组件/API 专项审核

请对 PC Demo 额外检查：

- 是否读取并使用了 `front-spec/INDEX.md`。
- 是否按实际用到的组件读取 `api/Button.md`、`api/Table.md`、`api/Form.md`、`api/Input.md`、`api/Select.md`、`api/Modal.md` 等文件。
- 是否把 `pc.input.Button`、`pc.display.Table` 等语义组件正确映射为真实组件，而不是直接写成代码组件。
- 是否存在手写按钮、表格、表单、弹窗、加载、提示来模拟已有公司组件。
- 是否存在自造属性，例如 `variant="ai"`、`tone="glass"`、`primaryDanger`。
- 是否能说明未映射组件的替代方案和待确认责任。

### 3.1 按钮与图标专项审核

请逐项检查所有按钮，尤其是筛选区、表格行操作区、批量操作区、分页区、弹窗底部和抽屉底部：

- 是否使用 Unicode 字符、emoji、特殊符号模拟图标，例如 `⌕`、`↺`、`□`、`✓`、`×`、`↓`、`‹`、`›`。
- 图标尺寸是否与文字字号匹配，PC 常规按钮图标建议 14px 或 16px。
- 图标是否垂直居中，是否存在上浮、下沉、偏左、偏右。
- 图标和文字间距是否统一，建议 4px 或 6px。
- 同一区域按钮高度是否一致。
- 主按钮、次按钮、危险按钮、文字按钮的视觉权重是否清楚。
- 禁用按钮的文字和图标是否同步弱化。
- 是否存在不必要的按钮图标，导致页面显得杂乱。
- 表格行操作按钮是否过重、过大，是否比表格内容更抢眼。
- 查询、重置、查看、展开更多、清空选择、上一页、下一页等基础按钮是否可以只保留文字。

### 4. 信息密度 / 冗余 / 间距审核标准

请重点检查：

- UI 页面中是否出现 `Component / Token Usage Manifest`、设计规范执行摘要、需求摘要、状态覆盖清单、待补规则清单。
- 第一屏是否被说明文案、状态全集、标签、卡片或 AI 服务列表挤占。
- 是否每个页面只有一个主任务、每个操作区只有一个主按钮。
- 列表页是否控制在页头、筛选、表格、分页/状态、必要辅助信息等 3-5 个区块。
- 看板页是否控制在筛选、指标、图表、异常列表、口径说明等 4-6 个区块。
- 标签是否默认 1-3 个，是否存在为每个字段都加标签的问题。
- 按钮是否过多，低频操作是否收进更多。
- 状态是否被全量常驻展示，还是只在对应状态发生时呈现。
- 页面外层、区块间、区块内、按钮组、标签组、表单行间距是否稳定。
- 是否存在卡片套卡片、每个模块都加边框和阴影、普通区块过度包边。
- AI 工作台是否只展开当前服务，非当前服务是否只保留入口。
- 账户、授权、服务矩阵、检测结果、推荐结果是否同时占用主工作区。

### 5. P0/P1/P2 分级规则

P0 阻断型问题，必须本期修复：

- 影响核心流程。
- 影响业务规则。
- 影响用户完成主任务。
- 影响研发理解。
- 影响验收通过。
- 产品规则缺失。
- 关键状态缺失。
- 移动端主链路不可用。
- 组件/Token 错误导致核心流程不可完成或研发无法理解。
- PC 真实组件/API 使用错误，导致代码不可落地或研发无法复用公司组件。
- 冗余信息导致主流程不可判断，或研发无法识别页面主任务。
- 把 output-report 当作最终验收依据，导致人工验收责任缺失。

P1 体验效率问题，建议本期修复：

- 不阻断主流程，但影响用户理解。
- 信息层级不清。
- 操作路径偏长。
- 组件不统一。
- 页面结构影响效率。
- 移动端适配不佳。
- 大量未映射组件或随机 Token 影响理解、效率或研发复用。
- 页面可用但信息过载、区块过多、主次不清。
- Manifest、规范清单、需求摘要进入 UI，影响业务阅读。
- AI 工作台和宿主业务页面层级混乱。
- 筛选区、表格区、详情区、抽屉区间距影响效率。
- 大面积按钮图标错位、尺寸混乱、Unicode 图标混用，影响页面专业度和研发复用。
- 表格行操作、筛选区、批量操作区按钮视觉权重混乱，影响用户判断主次操作。
- 大量绕开 `front-spec` 已有组件，手写控件或自造 props，增加研发返工。

P2 体验细节问题，可后续优化：

- 颜色、间距、图标、微文案、局部一致性。
- 局部 Token 或组件状态不一致，但不影响主流程和研发理解。
- 局部间距、标签密度、按钮组间距、卡片密度问题。
- 局部按钮图标偏移、间距不一致、图标略大或略小，但不影响主流程。

### 6. 审核结论规则

请从以下 4 类中选择一种：

1. 建议通过：主流程可跑通，无 P0，无关键 P1，研发可理解字段、状态和操作。
2. 有条件通过：无 P0，有少量 P1，修复路径清楚。
3. 不建议进入研发：存在 P0，或多个 P1 叠加导致研发返工风险高。
4. 需补充产品规则后再审：字段、权限、状态、验收标准或 AI 使用边界不足。
