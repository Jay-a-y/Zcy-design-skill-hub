# 覆盖范围检查

本文件用于判断云随需求是否可以由当前 Skill 直接生成。

## 检查项

生成前至少检查：

- 业务对象是否明确。
- 用户角色是否明确。
- 项目阶段是否明确。
- 权限和可执行操作是否明确。
- 核心流程是否明确。
- 进入条件和完成条件是否明确。
- 是否存在正式 Figma 样板。
- 是否存在成熟页面模式。
- 是否存在 UDesign 映射。
- 是否需要项目插图；若无正式 Figma 插图，是否可使用 UDesign Empty 或云随中性占位兜底。
- 关键状态和异常路径是否已有规则。

## 覆盖结果

### 已覆盖

满足以下条件时视为已覆盖：

- 业务对象、角色、阶段、服务和目标明确。
- 核心流程可以映射到现有业务模型和页面模式。
- 已有正式 Figma 或成熟页面模式足以确定核心结构。
- UDesign 映射足以实现基础组件和交互。
- 若缺少的只是普通空状态或状态插图，可由 UDesign Empty / 云随中性占位兜底。
- 关键状态和异常路径已有规则。

处理方式：直接生成 Demo，并执行 P0/P1/P2 自检。

### 部分覆盖

满足以下条件时视为部分覆盖：

- 主流程和核心页面结构已覆盖。
- 缺少的是非核心视觉元素、普通空状态插图、局部说明、辅助信息密度或轻量交互。
- 缺失部分不会改变权限、风险判断、核心流程或结果决策。

处理方式：成熟部分直接生成；非核心缺失可使用云随内部补全策略，普通空状态可使用 UDesign Empty 或云随中性占位，并在交付中记录补全范围。

### 未覆盖/全新复杂模式

出现以下任一情况时视为未覆盖：

- 新增业务阶段或新角色工作台。
- 新首页骨架或全新核心流程。
- 新的高风险或不可逆操作。
- 新的风险判断机制、关键结果决策或复杂证据交互。
- 正式样板不足以确定核心布局和项目视觉。
- 插图本身承载核心业务含义、风险解释、结果判断或强品牌主视觉，且没有正式样板或项目资产。
- 角色、阶段、权限或数据来源存在实质冲突。
- 需求要求改变已有正式 Figma 的核心布局，但未明确设计变更。

处理方式：不要自行设计核心复杂页面，输出设计缺口。

## 云随内部补全策略使用边界

仅缺少非核心视觉元素、普通状态插图、局部信息组织或轻量交互时可使用云随内部补全策略。

缺少核心结构、权限、风险机制、关键流程或结果决策时，必须输出设计缺口。不得把 AI 自行猜测的内容当成已覆盖。
