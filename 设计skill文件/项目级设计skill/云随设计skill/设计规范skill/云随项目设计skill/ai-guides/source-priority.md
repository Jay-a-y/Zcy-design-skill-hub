# 来源优先级

云随需求生成时必须分开判断业务事实和设计事实。

## 业务事实优先级

本次已确认需求  
→ 云随业务模型和权限规则  
→ 已沉淀稳定业务规则  
→ 待确认事项

说明：

- 需求文档可以改变业务内容、字段、流程、文案和测试数据。
- 需求文档中的业务变化如果与 Skill 已沉淀规则冲突，必须标记为待确认或设计变更。
- 不得用 AI 猜测补齐关键权限、风险机制或阶段规则。

## 设计事实优先级

正式 Figma  
→ 云随成熟页面模式  
→ UDesign 基础组件  
→ UDesign Empty 状态插图  
→ 云随内部补全策略

说明：

- 正式 Figma 决定已有页面的布局、层级、尺寸、项目视觉和页面组合。
- 云随成熟页面模式用于组合和复用已沉淀结构。
- UDesign 提供基础视觉交互组件和框架实现，不覆盖正式 Figma。
- UDesign Empty 状态插图只用于缺少正式 Figma 插图的普通空状态、成功、失败、网络异常和维护中页面。
- 云随内部补全策略仅用于非核心、无正式样板、无成熟模式、无 UDesign 映射的细节；不读取外部通用基础设计 Skill。

## 需求与设计冲突

需求文档不能默认覆盖已有正式 Figma 中的布局和层级。

如果本次确实要改变已有设计，必须在交付中标记为“设计变更”，并说明：

- 原有正式 Figma 或成熟模式是什么。
- 本次要求改变什么。
- 改变是否影响核心流程、权限或结果判断。
- 是否需要设计师补充样板。

## 产品 Demo 的边界

产品 Demo 只提供功能、角色、阶段和流转依据，不作为正式视觉依据。产品 Demo 与正式 Figma 冲突时，视觉和布局以正式 Figma 为准；业务事实冲突时列为待确认。
