# 架构说明

云随项目级设计 Skill 的机制是：

```text
设计师建设与升级
→ 项目级 Skill 发布
→ 产品/研发提交需求和数据
→ AI 执行覆盖检查
→ 已覆盖生成 Demo
→ 部分覆盖非核心补全
→ 未覆盖输出设计缺口
→ 设计师补充样板或规则
→ 真实需求回归测试
→ 发布新版本
```

## 设计师建设与升级

设计师负责在发布或升级 Skill 前完成复杂度判断、核心 Figma 样板、业务规则、页面模式、UDesign 映射和回归测试。

L0～L3 是设计师建设和维护 Skill 的治理方法，不是产品经理或研发日常生成 Demo 的输入要求。

## 产品/研发日常使用

产品或研发提供 Skill、需求文档、业务诉求和测试数据。AI 在成熟范围内直接生成单 HTML Demo，并输出 P0/P1/P2 自检。

## 覆盖检查

AI 必须先判断需求是否已覆盖。缺少核心结构、权限、风险机制或关键流程时，输出设计缺口，不自行创造复杂页面。

## Skill 的边界

Skill 不是云随设计百科，也不是自动替代所有设计。它沉淀的是可复用的项目级页面规则、成熟模式和生成边界。

设计介入被前置到核心模式和复杂页面。AI 日常执行的是复用、组合、生成、评审和缺口识别。

## 集合页边界

内部 Skill 集合页未来只负责展示、下载、版本和反馈，不承担在线生成，不替代 AI 工具执行，也不承担云随业务规则判断。
